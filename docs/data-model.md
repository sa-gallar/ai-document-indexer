# Модель данных (Data Model)

В данном документе зафиксированы структуры входных и выходных данных для системы `AI Document Indexer`. Система спроектирована для обработки отсканированных документов через ИИ-модели и агрегации извлеченных данных в единый структурированный классификатор.

---

## 📥 Структура входных данных (Input Data)

Для каждого графического файла веб-приложение формирует и передает в конвейер ИИ следующий набор метаданных:

### 🏷️ Таблица входных параметров (с типами данных и обязательностью)

| № | Название поля | Тип данных | Обяз. | Описание / Ограничения | Пример заполнения |
| :--- | :--- | :--- | :---: | :--- | :--- |
| 1 | `file_path` | String | **+** | Полный путь к файлу, включая имя корневой и вложенных папок. Уникально идентифицирует каждый документ. | `/Archive_2026/Fund_12/Opis_3/Delo_45/scan_001.jpg` |
| 2 | `file_name` | String | **+** | Имя файла с расширением (выделяется из `file_path`). Допускаются буквы, цифры, дефисы и подчёркивания. | `scan_001.jpg` |
| 3 | `document_script_era` | String | **-** | Эпоха и орфография исходного документа. Настраивает OCR-модель ИИ. Варианты: `"pre-reform"` (дореформенная орфография до 1918), `"soviet"` (советская эпоха 1918–1991), `"modern"` (современная, по умолчанию). | `"pre-reform"` |
| 4 | `output_normalization_language` | String | **-** | Языковой стандарт для формирования финального JSON. По умолчанию `"ru-modern"`. Влияет на транслитерацию и современизацию текста. | `"ru-modern"` |
| 5 | `target_document_type` | String | **+** | Тип документа для сбора данных и извлечения метаданных (по умолчанию: `"census"`). Может быть `"census"`, `"birth_register"`, `"passport"`, `"other"`. | `"census"` |
| 6 | `separator_document_types` | Array (String) | **-** | Список типов документов, которые служат разделителями контекста в обработке (например, для разделения двух разных писем в одном сканированном документе). Может быть пусто. | `["census", "birth_register"]` |

**Легенда:** **+** = Обязательное поле | **-** = Необязательное поле

---

### Пример 1: Минимальная запись (обязательные поля)

## 📤 Результирующая структура: Фамильно-Адресный классификатор (Output Data)

Единая результирующая матрица, в которую последовательн�� агрегируются данные из всех обработанных файлов. Каждая строка представляет одного упоминаемого в документе человека.

### 🏷️ Таблица выходных полей (с типами данных и ключевыми признаками)

| № | Название поля | Тип данных | Ключевое | Обяз. | Описание алгоритма заполнения | Пример заполнения |
| :--- | :--- | :--- | :---: | :---: | :--- | :--- |
| 1 | `FIO` | String | **Да** | **+** | Извлеченное ИИ ФИО (или Фамилия и Инициалы), приведенное к современной орфографии и нормализованное. | `Иванов Иван Иванович` или `Иванов И.И.` |
| 2 | `Rank_Position` | String | Нет | **-** | Должность, сословие или воинское звание лица, упомянутое в документе. Может содержать дополнительную информацию (титул, звание). | `Старший унтер-офицер`, `Купец 1-й гильдии`, `Чиновник` |
| 3 | `Locality` | String | **Да** | **-** | Населенный пункт (город, село, деревня, хутор), связанный с лицом в документе. Нормализуется по современной географической номенклатуре. | `Москва`, `с. Петровское` |
| 4 | `Doc_Number` | String | Нет | **-** | Номер документа / распоряжения / архивного приказа. Извлекается в оригинальном виде из источника. | `№ 5`, `№ 4517-A` |
| 5 | `Doc_Date` | String | Нет | **-** | Полная дата документа (включая день, месяц и год). Формат: дд.мм.гггг (ISO 8601) или текстовый. | `10.05.1915` или `1915-05-10` |
| 6 | `Source_File` | String | Нет | **-** | Имя исходного файла, из которого извлечена запись (служит для отслеживания источника). | `scan_001.jpg` |

**Легенда:** 
- **Ключевое** = Поле используется для дедупликации и поиска дублей
- **+** = Обязательное поле | **-** = Необязательное поле
- Если поле необязательно и отсутствует — может быть `null` или опущено

## 🔁 Политика обновления и повторное сканирование файла (Update Policy & Re-scanning)

### Основные принципы:

1. **Уникальность по файлу:**
   - Каждая запись имеет уникальный идентификатор — комбинацию `file_path` + `FIO` + `Locality`.
   - Дублирование предотвращается на уровне агрегации.

2. **Atomic Update (Атомарное обновление):**
   - При повторном сканировании одного файла все записи, извлеченные из этого файла, **полностью перезаписываются**.
   - Старые данные из файла удаляются и заменяются новыми извлеченными данными.
   - Это гарантирует консистентность и актуальность данных.

3. **Обработка удаленных файлов:**
   - Если файл был удален из исх��дной директории, то записи, связанные только с этим файлом, также удаляются из классификатора.
   - Записи, присутствующие в нескольких файлах, остаются в классификаторе.

4. **Переименование или перемещение файла:**
   - Система использует **хеш содержимого файла (MD5/SHA256)** для отслеживания физического содержимого.
   - Если файл переименован или перемещен, но содержимое идентично (тот же хеш), система распознает это как **обновление пути**.
   - Старые данные о пути удаляются, новый путь записывается.

5. **Версионирование и логирование:**
   - Каждое изменение (добавление, обновление, удаление) логируется с меткой времени.
   - Для аудита доступен журнал действий (audit log).

### Пример сценария обновления:

**Итерация 1:** Файл `scan_001.jpg` содержит записи:
- FIO: "Иванов И.И.", Locality: "Москва"
- FIO: "Петров П.П.", Locality: "СПб"

**Итерация 2:** Тот же файл `scan_001.jpg` переобработан с улучшенной ИИ-моделью.
Новые извлеченные данные:
- FIO: "Иванов Иван Иванович", Rank_Position: "Купец", Locality: "Москва"
- FIO: "Петров Петр Петрович", Rank_Position: "Офицер", Locality: "СПб"
- FIO: "Сидоров С.С.", Locality: "Казань" (новая запись)

**Результат:** 
- Записи о Иванове и Петрове **полностью заменены** на новые версии (с улучшенной информацией о должностях).
- Добавлена новая запись о Сидорове.
- Старые сокращённые версии удалены.

## 🛡️ Обработка edge cases (Edge Cases)

| Ситуация | Обработка | Результат |
|---|---|---|
| Файл содержит пустой текст (blank page) | Запрос отправляется в ИИ, результат — пустой/null | Новых записей не до��авляется, файл помечается как "обработан" |
| Файл поврежден / не читаемый | Исключение перехватывается, файл пропускается, логируется ошибка | Файл добавляется в error log, обработка продолжается |
| Неправильный путь указан | Исключение FileNotFoundError, пользователю выводится сообщение об ошибке | Сканирование прерывается с предложением проверить путь |
| Файл переместился между сканированиями | По хешу содержимого определяется как существующий, путь обновляется | Данные сохраняют связь с тем же физическим файлом |
| ИИ-модель вернула null/пустой результат | Запись не добавляется в классификатор, событие логируется | Файл считается обработанным, но без извлеченных данных |

## 📝 Рекомендации для разработчиков

### При реализации хранилища:
- Используйте **UNIQUE constraints** на комбинацию `(file_path, FIO, Locality)` для предотвращения дублей.
- Сохраняйте **хеш содержимого** каждого файла для отслеживания переименований.
- Логируйте все операции CRUD с меткой времени для аудита.

### При экспорте данных:
- JSON: Кодировка UTF-8, поддержка кириллицы.
- CSV: RFC 4180, разделитель — запятая или точка с запятой (в зависимости от локали).
- XLSX: Используйте типы данных (дата, строка), форматирование для читаемости.

### При тестировании:
- Создавайте тестовые наборы данных, соответствующие структурам выше.
- Проверяйте обработку пустых файлов, поврежденных данных, edge cases.
- Тестируйте повторное сканирование и обновление записей.
