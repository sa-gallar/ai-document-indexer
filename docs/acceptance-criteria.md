# 🎯 Критерии приемки и оценка качества (Acceptance Criteria & QA)

Проект считается успешным только в том случае, если автоматизированный ИИ-разбор не уступает по качеству работе эксперта-архивиста. Документ определяет методологию тестирования, измеряемые метрики качества (QA) и технические ограничения системы.

---

## 1. Методология тестирования (Машинный vs Ручной разбор)

Проверка качества работы ИИ-индексатора проводится методом «слепого теста» по следующему алгоритму:
1. **Формирование контрольной группы:** Выбирается выборка сложных архивных файлов (сканов), которые ранее **не подавались** на вход ИИ и не использовались для настройки промптов.
2. **Формирование эталона (Ground Truth):** Профессиональный архивист вручную вычитывает эти файлы и составляет эталонную таблицу данных (выписывает все ФИО, адреса, должности, даты). Этот результат принимается за 100% точности.
3. **Машинный этап:** Те же файлы отправляются на обработку в разрабатываемую программу `AI Document Indexer`.
4. **Сравнение результатов:** Проводится автоматизированное перекрестное сравнение двух наборов данных для расчета целевых метрик.

---

## 2. Метрики успешности (Допуск к релизу)

Для успешного прохождения тестирования система должна строго удовлетворять следующим математическим и функциональным критериям:

### 📊 Качество извлечения данных (Метрики NLP)
* **Полнота извлечения (Recall) $\ge 95\%$:** ИИ должен обнаружить не менее 95% сущностей (ФИО и адресов) от общего объема сущностей, найденных человеком в эталоне. Пропуск фамилии критичен для генеалогического поиска.
* **Точность извлечения (Precision) $\ge 95\%$:** Доля корректно извлеченных сущностей среди всех, что нашел ИИ, должна быть не менее 95%. Объем ложноположительных срабатываний («галлюцинаций», когда случайный текст принят за сущность) не должен превышать 5%.

### 💾 Форматирование и целостность данных (Data Integrity)
* **Формат выгрузки:** Итоговые данные должны сохраняться строго в формате `CSV` в кодировке `UTF-8` с использованием стандартного разделителя (запятая `,`).
* **Стандартизация дат:** Все извлеченные даты приводятся к единому стандарту ISO `YYYY-MM-DD`. Если в документе указан только год, поле заполняется как `YYYY-00-00`.
* **Связанность данных:** В структуре данных должна поддерживаться строгая связь (через сквозной ID строки) между извлеченным ФИО, его должностью и адресом, упомянутыми в рамках одного контекста.

---

## 3. Границы проекта и ограничения (Scope & Constraints)

Чтобы метрики качества оставались релевантными, на входную контрольную группу накладываются следующие ограничения:
* **Качество сканов:** Разрешение изображений должно быть не ниже 300 DPI, текст должен быть читаем человеком без экстремальных усилий (исключаются файлы с критическими физическими повреждениями, замятиями или залитыми чернилами областями).
* **Тип шрифта:** Текущая версия системы валидируется строго на печатном (машинописном) тексте. Разбор рукописных фондов выносится в отдельный этап проекта с индивидуальными (более низкими) пороговыми метриками качества.

---

## 4. Нефункциональные критерии (Performance AC)

* **Производительность:** Время обработки одной страницы документа (OCR + LLM-анализ) в рамках пакетной загрузки не должно превышать в среднем 15 секунд.
* **Отказоустойчивость (Resilience):** Программа должна иметь встроенную политику повторных попыток (Retry Policy) для обработки сетевых ошибок или таймаутов API (например, ошибок `HTTP 429 Too Many Requests`). Сбой при обработке одной страницы не должен приводить к падению всего процесса пакетной индексации; ошибка должна логироваться, а процесс — переходить к следующему файлу.
